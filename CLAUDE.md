# Fabietti.xyz — Adaptive Research Analyst

## What This Repo Is

An adaptive research analyst workstation that produces expert-grade research reports as beautifully typeset PDFs. The system **automatically detects the domain** of any topic and adapts its entire pipeline — persona, research strategy, report structure, review criteria, and PDF template — to act as the best analyst for that field. A financial analysis gets an equity research analyst; a geopolitical question gets an intelligence analyst; a scientific topic gets a research scientist.

## Primary Workflow

**One command does everything:**

```
/report <topic>
```

This runs the full adaptive pipeline:
1. **Domain Classification** — detects the domain, selects analyst persona, writes `domain-profile.md`
2. **Scaffold & Brief** — creates project, generates domain-appropriate brief
3. **Research Plan** — domain-informed plan with appropriate methodology
4. **Deep Dives** — spawns parallel sub-agents with domain-specific instructions
5. **Synthesis** — writes report using domain-specific structure and style
6. **Review & Polish** — domain-specific quality checks
7. **Render & Deliver** — PDF with domain-appropriate template, emailed

Optional flags:
- `--template default|academic|financial|briefing` — override auto-selected template
- `--depth shallow|standard|deep` — controls research depth (default: standard)
- `--no-email` — skip email delivery

## Domain Profile

Every project gets a `domain-profile.md` file (auto-generated by Phase 0) that controls the entire pipeline:

- **Classification**: domain, sub-domain, analyst persona, report type
- **Report Structure**: custom section outline (replaces generic structure)
- **Deep-Dive Instructions**: domain-specific research guidance for sub-agents
- **Source Priorities**: ordered list of authoritative source types
- **Review Criteria**: domain-specific quality checks
- **Template**: which PDF template to use
- **Brief Sections**: domain-appropriate brief structure
- **Writing Style**: prose style guidance

## Manual Workflow (More Control)

For when you want to drive each step yourself:

1. `/new-research "Topic Name"` — scaffold the project folder
2. Edit `brief.md` — define the research question and scope
3. `/plan` — generates domain profile (if missing) + research plan from the brief
4. `/deep-dive <subtopic>` — investigate each subtopic (repeat as needed)
5. `/synthesize` — turn notes into a structured report draft
6. `/review` — critically review and improve the draft
7. `/render` — render report.md to PDF

## Key Commands

```bash
# Scaffold a new research folder
uv run python scripts/new_research.py "Topic Name"

# Render a report to PDF (auto-detects template from frontmatter)
uv run python scripts/render.py research/<project>/report.md

# Render with specific template
uv run python scripts/render.py research/<project>/report.md --template financial

# Render with custom output path
uv run python scripts/render.py research/<project>/report.md --output ~/Desktop/report.pdf
```

## Research Conventions

### Project Structure
Each research project lives in `research/<slug>/` with:
- `domain-profile.md` — domain classification and pipeline configuration (auto-generated)
- `brief.md` — research question, scope, objectives (domain-adapted)
- `notes/` — one file per source or subtopic (raw research)
- `report.md` — the final report (single source of truth, gets rendered to PDF)
- `output/` — generated PDFs

### Working with Research
- The system auto-generates `domain-profile.md` — you can edit it to override any classification.
- Always start by reading `brief.md` to understand scope.
- Keep notes organized: one file per subtopic or source in `notes/`.
- `report.md` is the single source of truth — it's what gets rendered.
- Use markdown features well: tables, footnotes, admonitions, blockquotes for citations.
- Every significant claim needs a source. Use footnotes (`[^1]`) or inline links.

## Writing Style Guidelines

Writing style adapts to domain (via `domain-profile.md`), but these universals always apply:

- **Clear, precise, analytical prose.** Not academic jargon, not blog-style fluff.
- **Lead with insight, not summary.** Every section should have a "so what."
- **Use concrete data.** Replace "significant growth" with the actual number. Replace "recently" with the date.
- **Acknowledge uncertainty.** Present conflicting evidence fairly. Don't paper over debates.
- **Every paragraph earns its place.** If you can't say why a paragraph exists, cut it.

## Report Markdown Features

Use these in `report.md`:

- **Tables**: for data comparisons (rendered with clean styling)
- **Footnotes**: `[^1]` for source citations
- **Admonitions**: domain-appropriate callout boxes:
  - `!!! insight "Key Insight"` — general insights (green)
  - `!!! thesis "Investment Thesis"` — financial reports (bold)
  - `!!! judgment "Key Assessment"` — intelligence briefings (amber)
  - `!!! finding "Key Finding"` — general findings (green)
  - `!!! warning "Risk Factor"` — warnings (amber)
  - `!!! danger "Critical"` — critical alerts (red)
- **Blockquotes**: for important citations (styled with accent border)
- **Code blocks**: fenced with language identifier for syntax highlighting
- **Horizontal rules**: `---` for section breaks

## Email Delivery

After rendering a PDF, it is automatically emailed via SMTP.

- **Default:** ON. Pass `--no-email` to `/render` or `/report` to skip.
- **Configuration:** set `EMAIL_RECIPIENT`, `SMTP_USER`, and `SMTP_PASSWORD` in `.env` (see `.env.example`).
- **Default sender:** `gabriele@fabietti.xyz` via Namecheap Private Email (SMTP: `mail.privateemail.com`).
- **Subject format:** `[Fabietti.xyz] <report title>`
- **Script:** `uv run python scripts/email_pdf.py <pdf-path> [--subject "..."] [--to "..."]`
- If sending fails, the render still succeeds — email is best-effort.

## PDF Rendering

The renderer (`scripts/render.py`) converts markdown → HTML → PDF:
1. Extracts YAML frontmatter (title, subtitle, date, author, template, report_type)
2. Converts markdown to HTML via Python-Markdown with extensions
3. Feeds HTML into a Jinja2 template (dynamically discovered from `templates/`)
4. Renders to PDF via WeasyPrint

Templates live in `templates/`. The CSS is in `templates/assets/style.css`.
